## 技术栈确认

**最终技术选型：**
- **后端框架**：FastAPI (Python) - 轻量级、高性能的REST API框架
- **前端框架**：React + TypeScript - 成熟的生态系统，适合复杂UI
- **UI组件库**：Material-UI (MUI) - 快速开发、良好的可访问性和主题支持
- **数据库**：PostgreSQL（生产环境）/ SQLite（开发环境）
- **认证方式**：JWT with refresh tokens - 无状态认证
- **存储层**：抽象层支持WebDAV + 本地挂载（SMB/CIFS、NFS）
- **部署方式**：Docker + Docker Compose
- **音频播放**：HTML5 Audio API，支持流媒体传输
- **状态管理**：React Context + useReducer（轻量级）
- **图表库**：Chart.js 或 ECharts（支持交互式图表）
- **测试框架**：pytest + httpx (后端)，Playwright (前端E2E)
- **数据库ORM**：SQLAlchemy
- **数据库迁移**：Alembic
- **API文档**：OpenAPI（从代码自动生成）

**架构决策：**
- **前端架构**：两个独立SPA应用
  - 用户界面：`/user` 路由下的独立应用
  - 管理界面：`/admin` 路由下的独立应用
- **API文档**：使用FastAPI自动生成OpenAPI 3.x规范
- **认证策略**：JWT with refresh tokens，短期访问令牌（15分钟），长期刷新令牌（7天）

## 原子提交策略

### 分支规划

- `main`（受保护）
- `develop`（开发主分支）
- `feat/phase-0-scaffold`（阶段0骨架）
- `feat/phase-1-models-api`（阶段1模型和API）
- `feat/phase-2-storage-ingestion`（阶段2存储和导入）
- `feat/phase-3-auth-playback`（阶段3认证和播放）
- `feat/phase-4-user-frontend`（阶段4用户前端）
- `feat/phase-4-admin-frontend`（阶段4管理前端）
- `feat/phase-5-analytics`（阶段5统计）
- `test/phase-6-unit`（阶段6单元测试）
- `test/phase-6-integration`（阶段6集成测试）
- `test/phase-6-e2e`（阶段6 E2E测试）
- `chore/phase-7-deploy`（阶段7部署）
- `release/vX.Y.Z`（版本发布）

### 提交消息示例

```
feat(auth): 添加JWT认证端点（登录、刷新、登出）
feat(db): 定义用户、内容和播放会话模型；添加Alembic迁移
feat(storage): 实现WebDAV客户端包装器和本地挂载适配器
feat(ingest): 实现内容导入管道和初始目录填充
feat(playback): 添加播放/暂停/定位端点；跟踪用户进度
feat(time-limit): 执行全局和每用户播放限制
feat(ui-user): 搭建用户界面React应用
feat(ui-admin): 搭建管理界面React应用
feat(analytics): 添加分析端点（每日/播放次数）
test(api): 添加认证流程单元测试
test(ui): 添加基于Playwright的UI测试（登录和库加载）
chore(doc): 更新API文档和开发者指南
```

### 提交规范

- 每个提交应该是原子性的、可独立构建和测试的
- 提交消息遵循Conventional Commits规范
- 每个PR应该只包含一个功能或修复
- 代码审查必须通过后才能合并
